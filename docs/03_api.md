## API Philosophy

> Think of the API in three layers:
>
> 1. **Ingestion layer** — pull data from GitHub into SQLite  
> 2. **Intelligence layer** — transform that data into vectors, clusters, and insights  
> 3. **Delivery layer** — stream results back to the frontend in real time

Every endpoint was designed around one rule: **never block the caller**. Heavy work (embedding, clustering, LLM calls) always runs in the background. The client gets an immediate response or a stream it can start reading right away.

## Authentication

**There is no API key or token required for any endpoint.**

The backend is a local-only development tool. Authentication is on the roadmap but not implemented. The GitHub token (`GITHUB_TOKEN` in `.env`) is used **server-side only** for GitHub API calls — it never passes through the client.

> ⚠️ **Do not expose this server to the internet** without adding auth. Every endpoint is fully public.

## Common Conventions

### Query Parameters vs Request Body

- `GET` endpoints use **query parameters** exclusively (e.g., `?repo=facebook/react`)
- `POST` / `PATCH` / `DELETE` endpoints that need a body use **JSON body** (`Content-Type: application/json`)
- `DELETE` endpoints that only need an identifier use **query parameters**

### The `repo` parameter

Almost every endpoint requires `repo`. It is always in the format `owner/repo` (e.g., `facebook/react`, `torvalds/linux`).

- As a query param: `?repo=facebook/react`
- URL-encoded when in a query string: `?repo=facebook%2Freact`
- In path segments: `/api/v1/github/ws/sync/facebook/react`

### Trailing Slash

FastAPI enforces trailing slashes on router-mounted paths. The `GET /api/v1/issues` path will return a `307 Temporary Redirect` to `GET /api/v1/issues/`. Always call list endpoints **with** the trailing slash.

## Error Envelope

All errors return a consistent JSON structure:

```json
{
  "error": {
    "code": 404,
    "message": "Cluster not found"
  }
}
```

For unhandled exceptions (500):
```json
{
  "error": {
    "code": 500,
    "message": "An unexpected system fault occurred."
  }
}
```

FastAPI's own validation errors (422 Unprocessable Entity) return the standard FastAPI format:
```json
{
  "detail": [
    { "loc": ["body", "repo"], "msg": "field required", "type": "value_error.missing" }
  ]
}
```

#### `POST /api/v1/github/sync`

**What it does:** This is the **most important endpoint in the system**. It does three things simultaneously:

1. **Starts** or **acknowledges** a background crawl of the GitHub repo (if not already running)
2. **Immediately** loads the current cluster cache from SQLite into memory and detaches from the DB session
3. **Returns** a `StreamingResponse` (Server-Sent Events) stream that the frontend reads in real time

Think of it like this: the background crawl is writing new data while the SSE stream is serving the **existing** cached data. The client gets instant results from the last sync while the new sync runs quietly in the background.

**Request body (JSON):**
```json
{
  "repo": "facebook/react"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `repo` | string | ✅ | `owner/repo` format |

**Response:** `text/event-stream` — Server-Sent Events stream (see [SSE Event Schema](#sse-event-schema))

**The stream lifecycle:**

```
Client connects
    ↓
Server: data: {"type":"status","payload":{"msg":"Accessing Matrix Cache..."}}
    ↓ (immediately, from DB cache)
Server: data: {"type":"cluster_found","payload":{...}}   ← repeated N times
Server: data: {"type":"cluster_found","payload":{...}}
    ...
    ↓
Server: data: {"type":"complete","payload":{"total_issues":2436,"total_clusters":764}}
    ↓
Stream ends. Background crawl continues...
```

If no clusters exist yet (first-ever sync):
```
Server: data: {"type":"status","payload":{"msg":"Indexed 0 issues. Synthesis in progress..."}}
Server: data: {"type":"complete","payload":{"total_issues":0,"total_clusters":0}}
```

**Concurrency guard:** If a sync for the same repo is already running, the second request does **not** start another background crawl — it just serves the cache. This prevents the double-click problem.

**Critical implementation detail:** The DB session is **closed before streaming begins**. This is not optional — holding it open during the SSE stream blocks the connection pool for all other requests. The cluster data is copied into plain Python dicts (`cluster_snapshots`) before the session closes and the `StreamingResponse` generator runs in the freed context.

**Example:**
```bash
curl -N -X POST http://localhost:8000/api/v1/github/sync \
  -H "Content-Type: application/json" \
  -d '{"repo": "facebook/react"}'
```

```
data: {"type": "status", "payload": {"msg": "Accessing Matrix Cache..."}}
data: {"type": "cluster_found", "payload": {"cluster_label": 0, "insight": "React rendering alignment issue...", ...}}
data: {"type": "cluster_found", "payload": {...}}
...
data: {"type": "complete", "payload": {"msg": "Matrix sync complete.", "total_issues": 2436, "total_clusters": 764}}
```

### GitHub — Cluster Operations

#### `GET /api/v1/github/cluster/{id}`

**What it does:** Returns the full detail of a single cluster by its DBSCAN label (integer). Used when clicking into a cluster from the Intelligence tab or from a direct URL (`/cluster/:owner/:repoName/:id`).

**Path parameters:**

| Param | Type | Description |
|---|---|---|
| `id` | integer | DBSCAN cluster label (≥ 0) |

**Query parameters:**

| Param | Type | Required |
|---|---|---|
| `repo` | string | ✅ |

**Response `200`:**
```json
{
  "cluster_label": 42,
  "insight": "React rendering alignment issue identified — pattern: event, react.",
  "llm_summary": "Full LLM output text here...",
  "similarity_score": "87.3%",
  "issue_count": 7,
  "urgency": "High",
  "github_issue_numbers": [9046, 13400, 15821, 18002, 19210, 20401, 21055],
  "repo": "facebook/react"
}
```

**Response `404`:** `{ "detail": "Cluster not found" }`

**Note on `github_issue_numbers`:** This is the list used by the frontend to render the issue rows in `ClusterDetail`. These are GitHub's issue numbers (what appears in the URL `#9046`), not internal DB IDs.

---

#### `DELETE /api/v1/github/cluster/{cluster_id}/issue/{issue_number}`

**What it does:** Removes a single issue from a cluster's tracked list. This is a **manual curation** action — a maintainer can say "this issue doesn't belong in this cluster." The change persists in SQLite but will be overwritten on the next full re-sync (since clustering is recomputed from scratch).

**Path parameters:**

| Param | Type | Description |
|---|---|---|
| `cluster_id` | integer | DBSCAN cluster label |
| `issue_number` | integer | GitHub issue number to remove |

**Query parameters:**

| Param | Type | Required |
|---|---|---|
| `repo` | string | ✅ |

**Response `200`:**
```json
{
  "status": "removed",
  "cluster_id": 42,
  "issue_number": 9046,
  "remaining": 6
}
```

**Response `404`:** If cluster not found, or if `issue_number` is not in the cluster's list.

---

#### `POST /api/v1/github/cluster/{cluster_id}/reanalyze`

**What it does:** Queues a background LLM re-analysis of a single cluster. The current `summary_insight` and `llm_full_analysis` fields are regenerated using the actual issue texts from the DB (not mocked). Returns immediately with a `queued` status — the re-analysis runs asynchronously.

**Path parameters:**

| Param | Type | Description |
|---|---|---|
| `cluster_id` | integer | DBSCAN cluster label |

**Query parameters:**

| Param | Type | Required |
|---|---|---|
| `repo` | string | ✅ |

**Response `200`:**
```json
{
  "status": "queued",
  "cluster_id": 42,
  "message": "Re-analysis running in background"
}
```

**Response `404`:** `{ "detail": "Cluster not found" }`

**What happens in background:** Fetches the 10 most recent issues in the cluster from `IssueModel`, builds context strings, calls `llm.generate_cluster_insight()`, and commits the new insight to `ClusterModel`. Typically completes in 2–5 seconds if Groq API is responsive.

### GitHub — Triage Management

Triage is the maintainer's annotation layer on top of an issue. You don't modify GitHub — you store your own notes, priority, and flags in the local `issue_triage` table.

#### `GET /api/v1/github/triage/{issue_number}`

**What it does:** Fetches triage metadata for a specific issue. Returns a default "blank" triage object if no triage record exists yet — this means the frontend never needs to null-check; it always gets a full object.

**Path parameters:**

| Param | Type | Description |
|---|---|---|
| `issue_number` | integer | GitHub issue number |

**Query parameters:**

| Param | Type | Required |
|---|---|---|
| `repo` | string | ✅ |

**Response `200` (with existing triage record):**
```json
{
  "issue_number": 9046,
  "repo": "facebook/react",
  "priority": "p1",
  "triage_status": "triaged",
  "bookmarked": true,
  "pinned": false,
  "locked": false,
  "linked_pr": "https://github.com/facebook/react/pull/29100",
  "notes": "Root cause confirmed: event propagation order in synthetic events."
}
```

**Response `200` (no triage record — default):**
```json
{
  "issue_number": 9046,
  "repo": "facebook/react",
  "priority": null,
  "triage_status": "needs-triage",
  "bookmarked": false,
  "pinned": false,
  "locked": false,
  "linked_pr": null,
  "notes": null
}
```

**What it does:** Creates or updates a triage record. This is an **upsert** — if the record doesn't exist, it's created; if it does, only the fields you provide are updated (the rest are left unchanged). All fields are optional.

**Path parameters:**

| Param | Type | Description |
|---|---|---|
| `issue_number` | integer | GitHub issue number |

**Query parameters:**

| Param | Type | Required |
|---|---|---|

```json
  "bookmarked": true,
```

### GitHub — Repository Management

#### `DELETE /api/v1/github/repo`

**What it does:** **Nuclear wipe.** Deletes all data for a repo from every storage layer:
- All rows in `issues` table for this repo
- All rows in `clusters` table for this repo
- Clears the in-memory `_vector_stores[repo]` entry
- Clears the in-memory `_sync_status[repo]` entry

**Does NOT delete the FAISS index files from disk.** Those would need to be manually deleted from `storage/vector_indices/`.

**Query parameters:**

| Param | Type | Required |
|---|---|---|
| `repo` | string | ✅ |

**Response `200`:**
```json
{
  "status": "flushed",
  "repo": "facebook/react"
}
```

**⚠️ This operation is irreversible. All indexed vectors and cluster analysis for the repo are gone.** A full re-sync is required to restore the data.

### GitHub — Spatial Matrix

#### `GET /api/v1/github/spatial`

**What it does:** Projects all FAISS vectors for a repo from 384 dimensions down to 2 dimensions using PCA (Principal Component Analysis). Returns `(x, y)` coordinates for every indexed issue, suitable for Canvas rendering in `SpatialMatrixView.jsx`.

Each point also carries its cluster assignment from the DB — allowing the canvas to color-code points by cluster.

**Query parameters:**

| Param | Type | Required |
|---|---|---|
| `repo` | string | ✅ |

**Requirements:** The repo must have been synced (FAISS index must be in `_vector_stores`). At least 3 vectors required for PCA to produce meaningful 2D coordinates.

**Response `200`:**
```json
{
  "repo": "facebook/react",
  "total": 2436,
  "explained_variance": [0.187, 0.092],
  "points": [
    {
      "id": 12345,
      "x": 0.4821,
      "y": -0.2034,
      "cluster_label": 42,
      "urgency": "High",
      "issue_number": 9046,
      "title": "Disabling or destroying event's target stops fu..."
    },
    {
      "id": 12346,
      "x": -0.1203,
      "y": 0.8821,
      "cluster_label": -1,
      "urgency": "noise",
      "issue_number": 13400,
      "title": "Boolean DOM properties coerce empty string to f..."
    }
  ]
}
```

**Field breakdown:**

| Field | Description |
|---|---|
| `explained_variance` | The fraction of total variance explained by PCA component 1 and component 2. E.g., `[0.187, 0.092]` means the 2D plot captures 27.9% of the total variance in the 384D space. |
| `points[].id` | Internal SQLite row ID (not the GitHub issue number) |
| `points[].x`, `points[].y` | PCA coordinates. No guaranteed range — the canvas normalizes these for display |
| `points[].cluster_label` | DBSCAN label. `-1` means noise (unclassified). ≥0 means a named cluster |
| `points[].urgency` | From `ClusterModel.urgency`. Noise points have `"noise"` |
| `points[].title` | Truncated to 80 chars to keep the payload size manageable |

**Response `404`:** `{ "detail": "No vector index found. Run a sync first." }` — thrown if the repo has never been synced or the server was just restarted with a cold `_vector_stores`.

**Response `422`:** `{ "detail": "Need at least 3 vectors for spatial projection." }`

**Performance note:** PCA on 2,436 × 384 matrix takes ~80ms on a single CPU core — fast enough to call on every page load. For repos with >50k vectors, this could reach 1–2 seconds.

### GitHub — Vector Index Telemetry

#### `GET /api/v1/github/vector-stats`

**What it does:** Returns full operational telemetry of the FAISS index for a repo. This is the data source for the entire `VectorIndexView.jsx` operational dashboard.

**Critically: this endpoint loads the FAISS index from disk if it's not already in memory.** This means it works immediately after a server restart — you don't need to re-sync to see your telemetry.

**Query parameters:**

| Param | Type | Required |
|---|---|---|
| `repo` | string | ✅ |

**Response `200` — when index exists:**
```json
{
  "repo": "facebook/react",
  "has_index": true,
  "indexed": 2436,
  "dimension": 384,
  "total_db_issues": 2460,
  "coverage_percent": 99.0,
  "memory_estimate_mb": 3.57,
  "index_file_size_mb": 3.54,
  "model_name": "all-MiniLM-L6-v2",
  "written_at": "2026-04-06T21:30:14.842310",
  "total_clusters": 764,
  "noise_count": 194,
  "clusters": [
    {
      "label": 42,
      "size": 31,
      "urgency": "Critical",
      "insight": "React rendering alignment issue identified...",
      "similarity_score": 91.2
    }
  ],
  "top_clusters": [...],
  "similarity_distribution": [
    { "range": "0-10%",  "count": 12 },
    { "range": "10-20%", "count": 45 },
    { "range": "20-30%", "count": 89 },
    { "range": "30-40%", "count": 201 },
    { "range": "40-50%", "count": 318 },
    { "range": "50-60%", "count": 289 },
    { "range": "60-70%", "count": 178 },
    { "range": "70-80%", "count": 102 },
    { "range": "80-90%", "count": 43 },
    { "range": "90-100%","count": 19 }
  ]
}
```

**Response `200` — when no index exists:**
```json
{
  "has_index": false,
  "indexed": 0,
  "dimension": 0,
  "total_db_issues": 0,
  "coverage_percent": 0,
  "memory_estimate_mb": 0,
  "index_file_size_mb": 0,
  "clusters": [],
  "similarity_distribution": []
}
```

**Field breakdown:**

| Field | How it's computed | What it means |
|---|---|---|
| `indexed` | `v_store.index.ntotal` | How many issue vectors are in FAISS |
| `dimension` | `all_vecs[0].shape[0]` | Vector dimensions — always 384 for MiniLM |
| `total_db_issues` | `COUNT(issues WHERE repo_name=?)` | Total issues in SQLite (including un-embedded ones) |
| `coverage_percent` | `indexed / total_db_issues × 100` | What % of DB issues have been vectorized |
| `memory_estimate_mb` | `indexed × dimension × 4 bytes / 1MB` | Estimated RAM usage (float32 per element) |
| `index_file_size_mb` | `os.path.getsize(*.index)` | Actual disk size of the FAISS binary file |
| `noise_count` | `indexed - sum(cluster sizes)` | Issues DBSCAN couldn't assign to any cluster |
| `similarity_distribution` | Pairwise dot product on 200 random vector samples | Shape of semantic similarity across the entire corpus |

**The similarity distribution** is one of the most useful operational signals:
- A distribution peaking in the 0–30% buckets = diverse issues, hard to cluster
- A distribution peaking in 50–80% = semantically cohesive corpus, good cluster quality
- A distribution peaking in 90–100% = potential duplicate issues dominating the repo

### GitHub — WebSocket Sync Feed

#### `WS /api/v1/github/ws/sync/{repo}`

**What it does:** Establishes a persistent WebSocket connection that broadcasts real-time background sync progress and GitHub new-activity notifications.

**Connection URL:**
```
ws://localhost:8000/api/v1/github/ws/sync/facebook/react
```

Note: `{repo:path}` means the path parameter captures slashes — `facebook/react` is passed as-is, not URL-encoded.

**Connection lifecycle:**

```
Client: WS CONNECT
Server: ACCEPT
    ↓ loop {
    Server: JSON message (every 2s when idle, every 0.5s while syncing)
    Server: [every 30s] polls GitHub Events API for new activity
    }
    ↓
Client: DISCONNECT
Server: graceful close, no error
```

**Message emitted every tick:**
```json
{
  "processed": 450,
  "total_repo": 2500,
  "is_syncing": true
}
```

| Field | Type | Description |
|---|---|---|
| `processed` | integer | Issues crawled and saved so far in this sync |
| `total_repo` | integer | Total open issues in the repo (from GitHub metadata) |
| `is_syncing` | boolean | `true` while background crawl is active |

**Message emitted when new GitHub activity is detected:**
```json
{
  "processed": 450,
  "total_repo": 2500,
  "is_syncing": false,
  "new_activity": true,
  "new_event_count": 3
}
```

| Field | Added when | Description |
|---|---|---|
| `new_activity` | New `IssuesEvent` or `IssueCommentEvent` detected | Signal to show "new issues detected" banner |
| `new_event_count` | Same | Number of new events since last poll |

**How GitHub Events are polled efficiently:**

The WebSocket uses HTTP conditional requests with `ETag` caching:
```
GET https://api.github.com/repos/{repo}/events
If-None-Match: {etag from last successful response}
```
- `204 Not Modified` → silently ignored (no GitHub API quota consumed beyond the request)
- New `ETag` returned → compare with stored ETag; if different, new events exist

Polling happens every 30 seconds (60 ticks × 0.5s) and only when `is_syncing=false`.

**Error handling:** Socket errors and `WebSocketDisconnect` are caught silently. The server does not crash on client disconnect.

### Issues — Ingestion & Listing

#### `GET /api/v1/issues/`

**What it does:** Lists all issues stored in the local SQLite database for a given repo. This is the **fallback display** mechanism — when DBSCAN produces 0 clusters (corpus too small), the Intelligence tab fetches this to show raw individual issues instead of an empty state.

**Query parameters:**

| Param | Type | Required | Default | Description |
|---|---|---|---|---|
| `repo` | string | ✅ | — | `owner/repo` |
| `limit` | integer | ❌ | 100 | Max issues to return |

Issues returned in descending `github_issue_id` order (newest first).

**Response `200`:**
```json
[
  {
    "id": 28869,
    "github_issue_id": 87,
    "title": "🎚️ Pitch shifter tremolo",
    "state": "open",
    "labels": null,
    "repo_name": "torvalds/AudioNoise"
  },
  {
    "id": 28871,
    "github_issue_id": 83,
    "title": "Prefer SVF over DF1/DF2 biquads",
    "state": "open",
    "labels": "enhancement",
    "repo_name": "torvalds/AudioNoise"
  }
]
```

**⚠️ Trailing slash required.** Call `GET /api/v1/issues/` not `GET /api/v1/issues` (the latter 307-redirects).

---

#### `POST /api/v1/issues/`

**What it does:** Manually ingest a single issue into the system. This is the **original ingestion endpoint** — predates the GitHub bulk sync pipeline. It embeds the issue immediately (synchronous), runs a duplicate check via FAISS similarity, and returns a priority score.

This endpoint is not called by the frontend today. It remains available for programmatic ingestion use cases (e.g., a webhook that calls this when a new issue is opened on GitHub).

**Request body (JSON):**
```json
{
  "title": "Timeout on checkout page",
  "body": "When users try to pay, the gateway times out returning 504."
}
```

| Field | Type | Required |
|---|---|---|
| `title` | string | ✅ |
| `body` | string | ✅ |

**Response `200`:**
```json
{
  "id": 1247,
  "title": "Timeout on checkout page",
  "priority_score": 0.7,
  "duplicate_count": 1,
  "similar_issues": [834]
}
```

| Field | Description |
|---|---|
| `id` | Internal SQLite row ID |
| `priority_score` | `0.5 + (duplicate_count × 0.2)` — higher score = more existing similar issues |
| `duplicate_count` | Number of existing issues with >85% cosine similarity |
| `similar_issues` | Internal DB IDs of the similar issues |

**Graceful degradation:** If embedding or FAISS fails, returns `{ priority_score: 0.5, duplicate_count: 0, similar_issues: [] }` — the issue is still saved to DB.

### AI Search

#### `POST /api/v1/ai/search`

**What it does:** The intelligence pipeline's crown jewel. Takes a natural-language question from a maintainer, embeds it, finds the 5 most semantically similar issues in FAISS, then sends those issues to Groq's Gemma 4 LLM to synthesize a conversational answer.

The full pipeline: `query text → embedding → FAISS top-5 → LLM → answer`

**Request body (JSON):**
```json
{
  "repo": "facebook/react",
  "query": "Why do hydration errors happen with server-side rendering?"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `repo` | string | ✅ | Must be a repo that has been synced this session |
| `query` | string | ✅ | Natural language question (any length) |

**Response `200`:**
```json
{
  "answer": "Based on the indexed issues, hydration errors in React's SSR often occur when the server-rendered HTML does not match the client-rendered output. As seen in Issue #18790, this typically happens when components use browser-only APIs like `window` or `localStorage` without proper SSR guards. Issue #21636 documents another common pattern where async data fetching causes state mismatches between server and client cycles.",
  "sources": [18790, 21636, 23401, 24055, 24890]
}
```

| Field | Type | Description |
|---|---|---|
| `answer` | string | Groq Gemma 4 synthesized answer (or keyword fallback mock if no API key) |
| `sources` | array of int | GitHub issue numbers of the 5 FAISS matches used as context |

**The `sources` array** is what the frontend renders as clickable `#18790` chips. Each chip links to `https://github.com/{repo}/issues/{id}`.

**Response `400`:** If the repo hasn't been synced in the current server session (cold `_vector_stores`):
```json
{ "detail": "Intelligence matrix for this repo not yet initialized. Please run a sync first." }
```

**This is a known cold-start limitation.** The vector store is loaded from disk by `GET /vector-stats` but not by this endpoint — you must call `/sync` at least once per server session before search works.

**Response `500`:** Any FAISS search or Groq API failure propagates as:
```json
{ "detail": "Search Engine Error: [description]" }
```

**Example:**
```bash
curl -X POST http://localhost:8000/api/v1/ai/search \
  -H "Content-Type: application/json" \
  -d '{"repo": "facebook/react", "query": "hydration errors in SSR"}'
```

### Clusters (Legacy)

#### `GET /api/v1/clusters/`

**What it does:** A legacy endpoint from the earliest version of the system. Re-runs DBSCAN live against the in-memory FAISS index of the `issues.py`-managed store (not the github.py pipeline). Returns cluster results with mock LLM insights.

**⚠️ This endpoint is vestigial.** It operates on a different `v_store` than the main sync pipeline (the one in `issues.py` module scope, not `_vector_stores` in `github.py`). In practice it returns an empty array unless the manual issue ingestion endpoint (`POST /api/v1/issues/`) has been used.

**Response `200`:**
```json
[
  {
    "cluster_id": 1,
    "issue_count": 3,
    "insight": "Authentication failure cluster detected — pattern in: auth, login.",
    "urgency": "Critical",
    "issues": [{"id": 12}, {"id": 45}, {"id": 89}]
  }
]
```

**Do not use this endpoint for the Intelligence tab.** Use `POST /api/v1/github/sync` instead.

**Request:** No parameters.

**Response `200`:**
```json

  "process": {
```

## SSE Event Schema

All SSE events from `POST /api/v1/github/sync` follow this envelope:

```
data: {"type": "<event_type>", "payload": { ... }}\n\n
```

The double newline (`\n\n`) terminates each event per the SSE spec.

### Event Types

#### `status`
Progress message for display in the UI spinner.
```json
{
  "type": "status",
  "payload": {
    "msg": "Accessing Matrix Cache..."
  }
}
```

#### `cluster_found`
One cluster result. The frontend pushes this into a buffer ref and merges into React state every 400ms.
```json
{
  "type": "cluster_found",
  "payload": {
    "cluster_label": 42,
    "insight": "React rendering alignment issue identified — pattern: event, react.",
    "llm_summary": "Full LLM analysis text...",
    "similarity_score": "87.3%",
    "issue_count": 7,
    "urgency": "High",
    "github_issue_numbers": "9046,13400,15821",
    "progress": "Loaded from DB cache"
  }
}
```

#### `complete`
Stream terminator. Sent once, after all `cluster_found` events.
```json
{
  "type": "complete",
  "payload": {
    "msg": "Matrix sync complete.",
    "total_issues": 2436,
    "total_clusters": 764,
    "repo": "facebook/react"
  }
}
```

#### `error`
Emitted if an exception occurs inside the SSE generator.
```json
{
  "type": "error",
  "payload": {
    "msg": "Database connection failed: ..."
  }
}
```

## WebSocket Message Schema

All messages sent over `WS /api/v1/github/ws/sync/{repo}` are JSON objects.

### Standard Progress Message (every tick):
```json
{
  "processed": 450,
  "total_repo": 2500,
  "is_syncing": true
}
```

### New Activity Message (only when new GitHub events detected):
```json
{
  "processed": 2436,
  "total_repo": 2436,
  "is_syncing": false,
  "new_activity": true,
  "new_event_count": 3
}
```

### Tick frequency:
- `is_syncing: true` → every **0.5 seconds** (fast for progress bar updates)
- `is_syncing: false` → every **2 seconds** (idle polling, minimal overhead)
